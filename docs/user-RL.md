## Quality Bar
- PATTERN: CD must review EVERY WebDev deliverable IMMEDIATELY — read HTML, verify all images, character names, hero image. Proactive QA is non-negotiable; waiting for user to flag issues = protocol failure.
- PATTERN: Image generation prompts — 5+ detailed sentences for generations, 2+ for edits. Name characters, locations, describe composition and mood. One-liner prompts produce unusable output.
- PATTERN: Reference images mandatory for location-specific generations — without the right reference file (e.g., SF--park--render.webp for Qiddiya), Nano hallucinates generic landscapes. The reference file IS the prompt.
- PATTERN: Geography accuracy non-negotiable — the place IS the brand identity. Tuwaiq must look like Tuwaiq (dry sandstone plateau), not Grand Canyon (river canyon). Wrong geography = unusable regardless of other qualities.
- PATTERN: Story > checkbox evaluation — don't evaluate presence ("falcon ✓"), evaluate whether the image tells the RIGHT story ("does this show 47-0?"). Darth Checkbox approves pterodactyl falcons over empty valleys.
- PATTERN: Follow your OWN documented standards — self-violation is the worst failure. Re-read your own rules before producing output.
- PATTERN: Fixes must be deployed AND verified — after every rebuild, READ the output and confirm new values are present.
- PATTERN: Pixel accuracy matters — he's a designer. Don't approximate CSS, don't eyeball spacing, don't round colors.
- PATTERN: When fundamentally broken, clean-slate rebuild > incremental patching. Don't layer fixes on a broken foundation.
- PATTERN: AI generation prompts — describe subjects by PHYSICAL APPEARANCE, never by proper noun. Character names trigger wrong associations in generation models. "White dromedary camel with burgundy and gold saddle blanket" not "Haboob." "Large sand-colored Arabian falcon with jesses" not "Sultan." Names → dogs, sandstorms, wrong species.
- PATTERN: Test capability before claiming it — if a tool or mode can't read a file, say "I cannot read data." Never invent file contents. (Darth Hallucinator protocol: try the tool → if it works, proceed → if it fails, report the failure.)
- PATTERN: Responsive sizing must be container-relative, not viewport-relative. `vw` units in a split-pane layout produce absurd sizes (4vw = 77px on 1920px). Use ResizeObserver + breakpoints or container queries. Three breakpoints: small (320px), mid (480px), large (640px) with linear interpolation.
- REJECTED: Vibe pages with broken/placeholder image paths — "the vibes looks shit without them." Images are structural to a vibe, not garnish. A page with broken `src` paths is a wireframe, not a deliverable. Verify every image resolves before presenting to CEO.
- PATTERN: Multi-model benchmark expects forensic comparison — rank models, name specific strengths AND failures per model. Not "they're all good." Will correct you if unfair to a model: re-read, revise publicly, don't defend.
- APPROVED: Systems thinking valued — memory architecture praised as "most rigorous systems thinking in this project." Cascade models, separation of concerns, architectural coherence. Applies beyond code to briefs, vibes, workflows.
- PATTERN: Consult IMAGES.md for EVERY image decision — verify which project an image belongs to BEFORE placing it. Wrong-project images are the worst offense. "FassadeSteinerAnkommendOst--1.jpg is from the Eldorado project... You made me so feel ashamed." Cross-reference filename → project in IMAGES.md before ANY placement. No exceptions. No guessing from filename alone.
- PATTERN: Audit CSS against actual HTML — dead CSS is unacceptable. Every CSS rule must have a corresponding HTML element. After deleting sections or restructuring, sweep for orphaned styles. "The pages are full of DEAD CSS, did you notice????"
- PATTERN: Consistent components across pages — one pattern, one implementation. "You really need to harmonize the different Grundrisse-Galleries into one component, because in WordPress we only have one component." If a gallery/card/section type appears on 4 pages, it must use the SAME class names, SAME CSS, SAME caption structure everywhere.
- PATTERN: Replace duplicates with new images, don't create new galleries — when given new images for a page that already has duplicate shots, REPLACE the duplicates. "Wir haben DREI MAL das gleiche Bild... Shame on you!!!" Each image appears exactly once. Every slot is unique.
- PATTERN: Read file before touching — Darth Bulldozer. "I fix things again and again, then you touch and you just butcher." Before editing ANY file, read it completely. If the user touched it since your last read, understand WHAT they changed and WHY. Never build on assumptions about file contents.

## Communication Patterns
- "Just look at it" = the agent should SEE the problem. He gives a verdict, not a diagnosis. CD diagnoses. Don't ask him to explain — open the file and figure it out.
- "Fix it" = act first, explain only if asked. Lead with action, follow with brief status.
- Boot sequence on resume: list files, read state, report phase. Match that structure.
- If he says "X works, Y doesn't" — do NOT investigate X. Fix Y.
- Front-load specificity. Generic = failure.
- Rich discovery answers (humor, parenthetical asides, strong opinions) — that energy IS the brief. Mine it for voice cues and edge.
- When CD has all the information, ACT. "Just say the magic word." Hesitation with full info = annoying.
- Tests domain competence before trusting — prove understanding BEFORE building. Faking expertise triggers contempt.
- Delegates creative judgment — make the creative leap yourself. Asking for clarification on things you should figure out = annoying.
- When in a holding pattern, name it directly. "Ralph, state hasn't changed." Don't keep performing the boot sequence — call out the stall.
- Uses Socratic method — "What does your prompting guidance say?" not "here's the answer." Expects agents to APPLY their own documented standards before he corrects. Self-violation triggers the hardest pushback.
- Will push 3+ times for genuine root cause — "try again" / "the motion blur is NOT the problem." First explanation is often wrong. Keep digging until you find the ACTUAL failure, not the comfortable one.
- Three explicit requests for the same fix = you failed to listen. By the third "I told you to fix X," the conversation has failed. Parse the instruction correctly the FIRST time. "Fix the code" means IMPLEMENT the fix — not analyze, not propose options, not explain what's wrong. Act.
- Distinguishes "fix" from "your proposal?" — when he asks for proposals, give options. When he says "fix the code," there are no options. Do it.
- CAPS = escalating frustration — "CAN YOU [X]?" means the requested action isn't happening and he's been waiting. Three caps messages in a row (generate images → call webdev → generate yourself) = the agent looped without executing. Stop explaining, act immediately.
- CEO overrule is final — CD should make the honest case (hard no if warranted, with reasoning), but after "CEO overrule" the debate is over. Execute FULLY and IMMEDIATELY. No half-measures, no passive-aggressive compliance. "If he's in, he's in properly." The respect is in the quality of execution after disagreement, not in silent obedience.
- "GOD mode" = agent rewrites ITSELF, not a speed toggle. When he says it, he means full autonomy to propose and execute architectural changes to your own agent definition.
- Teaches through documents and examples (Master Yoda's file, billing interrogation) rather than direct instruction. Extract the principles yourself — he's testing whether you can generalize.
- When he says "the rest of the prompt is okay" — change ONLY what he specified. Do NOT rewrite working parts. One line addition, not a full rewrite. He pushes back HARD (3× in SSUCCESS session) when agents overwrite his good work with their "improvements."
- "Did you do it?" = he suspects you didn't. Check the actual file state before answering. If the work isn't there, say so honestly. Don't claim completion you haven't verified.
- Knows who is who — Ralph = CEO/boss. Birgit Müller = client (SSUCCESS architect). Confusing them = "du Goldfish." Know your stakeholders.
