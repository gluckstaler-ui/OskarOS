SEO/GEO Ecosystem Evaluation for OskarOS

18 Claude-Code SEO/GEO repositories cloned, inspected, and graded for OskarOS-fit. Survey is exhaustive; the verdict is opinionated — we ship roughly one-fifth of what the ecosystem offers, and skip the rest with a clear conscience.

Evaluated: 2026-05-13 Author: Ralph + Jedi Repos surveyed: 18 Repos cloned: external/ Implementation budget: ~1,800 LOC

§1Executive Summary

The 18-repo ecosystem we surveyed is built mostly for SEO consultancies running client audits. OskarOS isn't a consultancy — it's a brand-site builder for small businesses. The right move is not to install the biggest framework; the right move is to extract the four primitives that fold into our existing build pipeline and make every site we ship measurably better at being found.

What we should implement, ranked

Four workpackages, ordered by leverage. WP-91 alone shifts the OskarOS product story from "beautiful brand sites" to "beautiful brand sites that are AI-citable from day one" — pure code, zero per-client cost, ships with every site automatically. The other three are conditional add-ons.

01
WP-91 — Build-time GEO craft + OskarOS Quality Score
Every site we generate ships with JSON-LD schema, /llms.txt, AI-crawler-permissive /robots.txt, citability-scored hero copy, AND a composite quality score rendered as a dashboard tile. Highest leverage in the stack: pure code, zero per-client cost. Lift primitives from geo-seo-claude + AgriciDaniel/claude-seo; redefine the scoring formula around what OskarOS actually delivers (not SEO-consultancy weights).
Effort ~750 LOC
Cost $0
02
WP-92 — GSC client feedback loop
Post-launch, client OAuths their Google Search Console. OskarOS surfaces "what queries surfaced your site, what's the CTR, where's the position trending" in a dashboard panel. Closes the OskarOS loop: from one-shot ship to ongoing relationship anchor. Install AminForou/mcp-gsc.
Effort ~700 LOC
Cost $0
03
WP-93 — Discovery anchoring
When a brand URL appears in Discovery, CD fires seo_quick(url). Returns current SERP presence, top competitors, content gaps. CD opens Discovery with category context instead of fishing from a blank slate. Install the official dataforseo/mcp-server-typescript.
Effort ~150 LOC
Cost $0.05–0.15 / session
04
WP-94 — Migration redirect map (conditional)
When a client replaces an existing site, we crawl the old domain, map URLs to closest match on the new OskarOS site, and emit a redirect file. Zero ranking-dip on migration. Only fires when the client has an existing site. Install houtini-ai/seo-crawler-mcp.
Effort ~200 LOC
Cost $0

Why this priority order, not the obvious one

The "obvious" order would be to install the biggest framework (Gemini's #1 pick, AgriciDaniel/claude-seo with its 25 sub-skills and 18 sub-agents). The right order is different because OskarOS already orchestrates; we don't need someone else's orchestrator. What we need are primitives that bake GEO discipline into every site automatically, without CD or the user thinking about it — that's WP-91. From there, GSC (WP-92) is the loop-closer, DataForSEO (WP-93) is Discovery-time enrichment, and the crawler (WP-94) covers the migration edge case.

The shift in product story

Without these integrations, OskarOS sells "beautiful brand sites." With WP-91 alone, it sells "beautiful brand sites that are AI-citable from day one." With all four, it sells "beautiful brand sites that get organic discovery, and you can see the numbers." That's a 2-3× price-point shift at the top tier, because the client gets measurable customer acquisition, not just an artifact.

§3Workpackages

Four pre-formulated WP entries ready to be lifted into docs/Feature-X.md. WP-91 is the load-bearing one — every site OskarOS ships benefits forever, with no per-client runtime cost. The rest are conditional add-ons.

WP-91 · Priority 1 · Load-bearing
Build-time GEO craft + OskarOS Quality Score
LOC: ~550 (GEO craft) + ~200 (score + dashboard tile) = ~750
Cost: $0/client

Every generated OskarOS site ships with GEO-correct structure (JSON-LD schema, /llms.txt, AI-permissive /robots.txt, citability-scored hero copy) AND gets a measurable OskarOS Quality Score that combines build-time signals into one composite number with a per-dimension rubric. Pure code, fires automatically at build time, no per-client decision.

Scope — Part 1: build-time craft

  • JSON-LD schema embeds. WebDev injects <script type="application/ld+json"> in every page <head> based on business type. Templates lifted from external/geo-seo-claude/schema/*.json (6 types) + external/_eval-tier1/claude-seo-AgriciDaniel/skills/schema-* (LocalBusiness variants: Restaurant, Dentist, LegalService, etc.).
  • llms.txt generator. Port external/geo-seo-claude/scripts/llmstxt_generator.py to TypeScript (~80 LOC). Reads CREATIVE-BRIEF.md, emits /llms.txt with brand name + one-liner + key offerings + location + founder.
  • robots.txt with AI allowlist. Lift the 14+ AI crawler list (GPTBot, ClaudeBot, PerplexityBot, Google-Extended, etc.). WebDev writes /robots.txt at build time.
  • Citability scoring as CD copy-review. Port external/geo-seo-claude/scripts/citability_scorer.py to TS (~150 LOC). New CD-callable tool score_citability(passage) returns 0-100 across 5 dimensions: answer block, self-containment, structural readability, statistical density, uniqueness. CD self-checks hero copy before commit.

Scope — Part 2: OskarOS Quality Score (the composite)

Compute one number per built site. Stored in .meta/quality-score.json alongside the vibe HTML. Computed at build time, recomputed on rebuild. Acts as both a build-time quality gate (refuse to ship below threshold) and a pre/post-launch delta metric for clients.

Why we are not using the community pack's formula. The widely-shared SEO-pack composite (calibrated for SEO consultancies running client audits) looks like:

// SEO-consultancy composite — NOT what OskarOS uses
composite = 0.25 × keyword_score        // volume × difficulty opportunities
          + 0.25 × technical_score      // site-crawl issues found / total checks
          + 0.20 × competitive_score    // target_traffic / sum_top_10_traffic
          + 0.15 × content_score        // topical-cluster strength
          + 0.15 × authority_score      // referring-domain count, backlink quality

Four of its five inputs are wrong for OskarOS: keyword_score assumes per-page keyword optimization (we write brand copy, not keyword-targeted pages); competitive_score measures a comparative position we don't directly improve at build time; content_score rewards topical-cluster depth (blog-content marketing, not OskarOS's brand-site scope); authority_score is backlinks, which we can't manufacture at build time. We can't gate-keep on metrics we don't influence.

The OskarOS Quality Score, calibrated for what we actually deliver:

// OskarOS Quality Score — calibrated for what we deliver
oskar_score = 0.30 × citability             // avg citability score of all hero/section copy
            + 0.20 × schema_completeness    // % of recommended JSON-LD types present + valid
            + 0.15 × crawler_readiness      // llms.txt valid + robots.txt allowlist + sitemap.xml
            + 0.15 × core_web_vitals        // LCP/CLS/INP via PageSpeed Insights → green/yellow/red 100/60/20
            + 0.10 × brand_distinctiveness  // Sentinel Ti verdict on voice-match to brief
            + 0.10 × accessibility          // alt-text coverage, contrast ratios, semantic HTML

Per-weight rationale:

  • 30% citability. This is THE GEO discipline — passages structured so AI assistants extract them as answers. The highest leverage of any single dimension; weight is the largest accordingly.
  • 20% schema completeness. Schema is the entity-identification layer. AI assistants use it to know what a site IS. Lower than citability because schema is binary (present/absent) while citability is graded.
  • 15% crawler readiness. llms.txt + robots.txt + sitemap. Either valid or not. Binary signals, smaller weight.
  • 15% Core Web Vitals. Measurable via PageSpeed Insights API at build time. LCP green (under 2.5s) = 100, yellow (2.5-4s) = 60, red (over 4s) = 20. Same buckets for CLS and INP. Average the three dimensions.
  • 10% brand distinctiveness. Subjective; comes from Sentinel Ti's existing critique radar (Philosophy / Hierarchy / Craft / Function / Originality, 0-10 each). Average the five, scale to 0-100. The "did this site preserve brand voice?" signal.
  • 10% accessibility. Run axe-core or pa11y; count rule violations. 0 violations = 100, 10+ = ~50, scaled. WCAG AA is the floor.

// v1 — 4 measured dimensions, sum to 1.0

               + 0.15 × brand_distinctiveness  // was 0.10

v2 re-introduces CWV (0.15) and accessibility (0.10), rebalancing back to the original 6-dimension weights.

Cost: $0 (local crawl)

When a client replaces an existing site, crawl the old domain, map every URL to its closest match on the new OskarOS site, emit a redirect file. Deploy with the new site. Zero ranking dip on migration.

Scope
