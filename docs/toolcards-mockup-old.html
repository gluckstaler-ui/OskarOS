Archetype 2 — Diff cards

Same diff renderer, opposite verbs. Both tools share the diff visual primitive but mean different things.

apply_patch WRITE

surgical change · agent makes the edit in place

CD calls apply_patch(target, edit) with one of six edit kinds: css-var-set, text-replace, attr-set, class-toggle, delete, insert.

Refuses <script> selectors. Records the diff so the change is revertable from the card foot.

When: trivia too small for build_vibe. Fix a typo, tighten a CSS var, swap an attribute. The edit happens immediately; the card is the receipt.

vibe_diff READ

comparison · system-fired on Director-OFF

The system fires this card automatically when the user turns Director Mode OFF and the changes are applied. Shows the text changes (per-selector, additions/deletions) so both user and CD can see what landed in this commit. Not CD-callable for this purpose — the director_save event drives the render.

If the same Director-OFF commit included image swaps, a separate hotswap (Swapped) card fires alongside this one — text changes in vibe_diff, image changes in the swap card. Two cards, one apply.

v1 spec lock: only since='last-build' works on the manual call (kept for CD's pre-build sanity check).

When (system-driven): Director Mode toggled OFF + apply. When (CD-driven, optional): pre-build sanity check.

Surface placement — overlay vs chat scroll latest · 2026-05-06

The tier system above tells you HOW each card is shaped. This section tells you WHERE it lives. Two surfaces total: chat scroll (default — every card is conversation record) + live overlay (mounted in ChatPanel, anchored top-right corner; cards flow horizontally side-by-side, wrap to vertical stack when out of space). Snackbars and the Sentinel feedback sub-tab are separate surfaces and don't count here.

This section supersedes the tool → surface map above where they disagree. Specifically: thread_history + claim_orphan chips are deleted entirely (debug noise); vibe_diff downgraded to snackbar (single-fire, low-volume); generate_image moved to inline (one-shot lifecycle, identical to image upload); submit_critique moved to the feedback sub-tab (out of chat/overlay scope); ask_user variants C (freeform) and F (multi-select) deleted (overlap with chat box and question-form respectively).

Criterion for overlay
Auto-updates over time AND would be a shame to scroll past. Single-fire receipts (hotswap, apply_patch, generate_image, screenshot) fail the first test and stay inline. Sentinel critique fails the first test too — single produce-and-commit, lives in feedback sub-tab.

Why the asymmetry: builds are events — once seen, the inline receipt is the durable record. TodoWrite is a shared visibility, single-writer queue (both parties see it; only CD writes — user contributions flow through chat, CD encodes them) — hiding it would lie to both parties about agent state. Minimized form sits as a single-line strip anchored to the overlay edge: ✦ Queue · 3 / 7 · CD on item 2 — click to expand.
Implementation impact (for JC)
  • Tier 2 build cards render INLINE as receipts AND clone a compact mirror row into overlay while running. Overlay-mirror primitive: <BuildOverlayRow jobId thumb id label timeline eta /> — same fields, no card chrome. Snaps out on terminal state.
  • Tier 3 TodoWrite panel renders in overlay ONLY (not inline). Originating agent-message anchors it; overlay is the live surface. Minimize toggles between full panel and single-line strip.
  • New container: <LiveOverlay /> — mounted in ChatPanel, anchored top-right corner. Layout: horizontal-first with wrap. flex-row + flex-wrap: wrap; newest card closest to the corner, older pushed left; wraps down when horizontal space runs out. 1 build + 1 todo on wide desktop = side-by-side; 4+ in-flight on narrow viewport = stacked. Visible only when ≥1 card has live content. Same theme tokens as ToolCard (theme-radius, 12-16px padding).
TodoWrite write authority — CD-only (locked)
User has READ access to the panel (sees the queue, sees items flip pending → in_progress → completed). User does NOT have direct write access. User contributions flow through normal chat — "add: redo vibe-3 hero" — and CD encodes the request as a todo on its next turn.

Result: no bidirectional MCP tools, no todo_user_change event, no race conditions. Single-writer store. Chat is already the channel for user→CD intent.
Storage — ## Todos section in SESSION.md
No new file. Todos live as a section in the existing SESSION.md, sibling to the existing ## Workflow State. Same checkbox idiom CD already maintains for workflow. Minimum-changes principle: reuse the file CD writes to anyway, reuse the grammar Sage already parses, no new write path, no new parser file.

Status mapping
[ ] → pending  ·  [/] → in_progress  ·  [x] → completed (paired with ~~strikethrough~~)
Position + parser
  • Section sits near the top of SESSION.md, just below ## Workflow State. Position is fixed — CD rewrites the body, never moves the heading.
  • Each - [x] content / - [ ] content / - [/] content line = one todo.
  • Checkbox glyph is the status source of truth. Strikethrough is cosmetic.
  • Order = priority. Anything not matching the checkbox-line shape is ignored.
Lifecycle
  1. CD calls TodoWrite({todos: [...]}) — full-list replace.
  2. Orchestrator find-replaces the ## Todos section body in SESSION.md + emits todos_updated on the bus.
  3. <LiveOverlay /> subscribes → re-reads section → re-renders panel.
  4. On session resume / agent respawn (Order 66, dev reload, crash): LiveOverlay reads the section from SESSION.md on mount. State survives because SESSION.md is already persisted.
User-add path uses the same storage: user types "add: redo vibe-3 hero" → CD's next turn calls TodoWrite with the appended item → orchestrator rewrites the section + broadcasts → panel updates. No second storage path. Single source of truth.

Universal user-input field — every discovery card carries it (locked 2026-05-07)
Every card surfaced during Discovery / Junior Pass / Vibes phases carries a textarea at the bottom labeled "Thoughts, comments, anything else?" Submitting the card posts the textarea content as a normal user message in chat, alongside the structured selection. CD reads both — the structured pick AND the freeform context.

Why universal: the user always has more to say than the form captures. A structured selection without a freeform escape hatch forces them to either stuff context into a label they don't control, or break out of the card flow entirely. Both are friction. The textarea is cheap to add and load-bearing for honest feedback.

No card ships without it. Direction-cards, moodboard, descent_selection, confirm_understanding, and any future Phase 1–3 card all compose the same <DiscoveryCardFooter /> primitive (per WP-73).
