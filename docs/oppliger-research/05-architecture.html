Kapitel 05 — Server-Administration

Wie Oskar den Server betreibt — Setup, Funktionsumfang, reale Grenzen

Oskar läuft auf dem Mac mini im Keller und sitzt als leichter MCP-Agent auf jedem Praxis-Laptop. Der Server trägt Admin, 24/7-Jobs und die Audit-Spur. Die Laptops bringen den Assistenten dorthin, wo gearbeitet wird — an Mail.app, Finder, Calendar, das lokale OneDoc-Tab.

Niemand sitzt am Keller-Mac — er ist headless, im Untergeschoss, hinter einer Tür. Die vier Laptops (Robert, Gabriela, Bettina, vierte Kollegin) plus die Empfangsstation sind die Arbeitsstellen. Oskars Aufgabe ist zweiteilig: den Keller-Mac unsichtbar-aber-administriert zu machen (Backups, Updates, FileMaker, Zertifikate) und gleichzeitig an den Arbeitsplätzen in den Apps zu sitzen, in denen die Praxis tatsächlich arbeitet. Das eine ohne das andere wäre eine halbe Antwort.

Hebelt den "Server aus der Ferne administrieren"-Pitch aus.

Der Keller-Mac läuft sowieso 24/7, hat die feste IP, hostet FileMaker Server. Er trägt den kanonischen Oskar — Zustand, Audit, Scheduling, Admin-Werkzeuge, alles, was niemand erreichen kann, wenn ein einzelner Laptop zugeklappt ist. Auf jedem Laptop sitzt zusätzlich ein leichter MCP-Agent (LaunchAgent, ~50 MB, autostart): derselbe Werkzeugkatalog wie auf dem Server, aber mit laptop-seitigen Aufrufen — mail.search via Mail.app-AppleScript, finder.read_dropped für das gerade abgelegte PDF, calendar.events_for für Roberts Kalender, onedoc.bookings_for aus dem geöffneten Tab. Die vier Laptops setzen ein Lesezeichen auf https://oskar.oppliger.ch/, der Server orchestriert, ruft bei Bedarf in den jeweiligen Laptop-Agent hinein. Eine Werkzeug-Definition, zwei Aufrufzonen.

Was Oskar administrieren kann — die volle Werkzeugfläche

Eine Vier-Ärzt:innen-Praxis hat dieselben administrativen Bedürfnisse wie jeder andere KMU-Server: Dienste müssen neu gestartet werden, Zertifikate erneuert, Backups gefahren und verifiziert, Benutzer:innen angelegt wenn neues Personal kommt und gesperrt wenn jemand geht, Software aktualisiert, Logs überwacht, Festplatten füllen sich, Mail-Queues stauen sich, SSH-Schlüssel altern. Oskar bietet einen umfassenden typisierten Katalog — keine kuratierte 25-Werkzeug-Auswahl, sondern die tatsächliche Breite, mit der ein Systemadministrator täglich arbeitet.

Was Claude tatsächlich nicht kann — ehrlich, ohne Ausschlussliste

Sieben echte Begrenzungen. Keine Datei-Pfad-Ausschlüsse. Oskar hat dieselbe administrative Reichweite, die ein Systemadministrator auf diesem Mac hätte — einschliesslich der FileMaker-Patientendatenbanken, des HIN-Mail-Stores, des Schlüsselbunds, der Backup-Volumes. Backup-Verwaltung bedeutet, die .fmp12-Dateien anzufassen; Restore bedeutet sie zu schreiben; Integritätsprüfung bedeutet sie zu lesen; Performance-Analyse bedeutet hineinzuschauen. Oskar dort auszusperren würde den Sinn eines Admin-Assistenten zunichtemachen. Die Grenze liegt in der Werkzeug-Semantik (was jedes Werkzeug tut, was es auditiert) — nicht in den Dateien, die es sehen kann.

GrenzeWarumAbhilfe
Hardware-Physik Kernel-Aufhänger, Boot-Fehler, Hardware-Defekte, FileVault-Boot-Screen, Stromausfälle — keine Software innerhalb des Macs kann den Mac retten, wenn der Mac selbst tot ist. Robert wohnt an derselben Adresse wie die Praxis (Hardturmstrasse 202). Eine Treppe runter, Stecker ziehen, Stecker rein — das ist die physische Recovery-Operation. M-Series Mac mini auf APFS verkraftet Stromausfälle problemlos (transaktionales Filesystem, SSD braucht kein Spin-down). Dokumentierte physische Recovery-Anleitung in practice-context.md.
Kontinuierliches Monitoring Claude sitzt nicht stundenlang vor den Logs und wartet auf Anomalien. Aufruf-Latenz ist 2–15 Sekunden — nicht "mein Log endlos tailen." Prometheus + Alertmanager (oder Better Stack / Datadog) läuft lokal auf dem Mac, schickt Alarme an Oskar. Claude reagiert auf Alarme, generiert sie nicht.
Sub-Sekunden-Incident-Response Wenn ein Brute-Force-Angriff SSH mit 100 Anfragen pro Sekunde trifft, muss die Blockade in Millisekunden passieren. Claude ist zu langsam. pf-Regeln + fail2ban übernehmen die autonome Reaktion. Claude prüft die resultierende Blocklist und aktualisiert die Policy im Nachhinein.
Rollback eigener Fehler Wenn Claude caddy.json falsch bearbeitet und neu lädt, kann Caddy abstürzen. Wenn er eine schlechte Migration auf eine FileMaker-DB fährt, können Daten beschädigt werden. Das Modell kennt die vorherige Version nicht automatisch. Schreibende Werkzeuge legen für ihre eigenen Config-Edits eine .bak-Datei daneben (fünf Zeilen Code, kein eigenes Snapshot-System). Für alles andere ist APFS + Time Machine die Rollback-Schicht — macOS macht stündliche Snapshots ohne Zutun, Time Machine konserviert die letzten 24 Stunden. fmaker.*-destruktive Operationen lösen vorher ein frisches fmaker.backup aus. Destruktive Operationen werden im Browser per Touch ID bestätigt.
Übergreifende Orchestrierung ohne expliziten Zustand Mehrstufige Workflows über mehrere Systeme (Registrar + DNS + Zertifikat + Caddy + Verifikation) brauchen Zustand, um Teilausfälle sauber zu behandeln. LLM-Gedankenketten sind keine Zustandsmaschinen. Lange Workflows werden in explizite Job-Definitionen auf der Festplatte kompiliert. Oskar führt den Job aus; Claude komponiert ihn, führt ihn aber nicht aus dem Gedächtnis Schritt für Schritt aus.
Praxisspezifische Intuition ohne Memory-Schicht Out-of-the-box erinnert sich Claude zwischen Sitzungen nicht, dass "der Keller im Juli feucht wird" oder "Bettina möchte Backups um 22:00, nicht um Mitternacht" oder "der Netzwerk-Switch fällt manchmal montags aus, nachdem die Putzkraft da war." Jede Sitzung startet bei null. Pro Deployment eine Datei ~/oskar/practice-context.md, die Claude beim Sitzungsstart liest. Wird von Robert / Bettina / Ralph gepflegt, wenn die Eigenheiten der Praxis sich ansammeln.
Hardware-Arbeit SSD wechseln, RAM tauschen, Netzteil ersetzen, Lüfter reparieren, Kabel einstecken, Mac mini umstellen. AppleCare + Kurier; Kalt-Reserve-Strategie, wenn Verfügbarkeit wirklich kritisch ist. Dokumentierte Wen-anrufen-Liste in practice-context.md.

Das ist die Liste. Keine Pfad-Ausschlüsse, kein "der Schlüsselbund ist tabu," kein "wir haben den Mail-Store eingemauert." Ein arbeitender Admin hat Reichweite über die ganze Maschine; Oskar bekommt dieselbe Reichweite, mit jeder Aktion im macOS Unified Log audit-geloggt und jeder destruktiven Operation per Touch ID am Browser bestätigt.

Was Oskar speziell macht, was andere Admin-Werkzeuge nicht machen

  • Natürlichsprachliche Komposition — Robert tippt "FMS scheint langsam, alle Backups ok?" auf Deutsch, Oskar wählt die richtigen Werkzeuge aus (fmaker.top_calls + tm.latest + restic.status + system.health), führt sie aus, fasst auf Deutsch zusammen.
  • Workflow-Komposition über Domänen — das 11-Schritte-macOS-Update unten kreuzt FileMaker, Time Machine, Software-Updates, Dienste und Benachrichtigung in einer einzigen Chat-Anfrage.
  • Audit-Spur sichtbar für den Arzt — kein undurchsichtiges Server-Log, das nur Ralph lesen kann. Robert öffnet "Letzte Aktionen" und sieht, was Claude wann mit welchem Ergebnis gemacht hat.
  • Snapshots vor Änderungen — jede Konfig-Bearbeitung erzeugt eine wiederherstellbare Vorher-Version. Niedrigere Einsätze für "Claude soll's mal probieren", weil Rollback ein Befehl ist.
  • Mehrsprachiges Reporting — Status-Meldungen in der Sprache, in der getippt wurde. Robert auf Deutsch, Ralph auf Englisch, Bettina in beidem.

Das 11-Schritte-orchestrierte macOS-Update — eine Chat-Anfrage

Das Beispiel, das seinen Platz verdient. Robert tippt mittags in den Chat: "Update macOS heute Nacht 22:30, vorher Backup, FileMaker nicht abschiessen."

Was Oskar ausführt — benannte Werkzeugaufrufe, jeder audit-geloggt

  1. 22:25 — fmaker.broadcast("Praxis-Server wird in 5 Minuten für Updates neugestartet. Bitte speichern und ausloggen.")
  2. 22:28 — fmaker.list_clients() → verifizieren, dass alle ausgeloggt sind
  3. 22:30 — fmaker.backup(dest="/Volumes/Backup/pre-update-2026-05-20/")
  4. 22:32 — fmaker.verify(file) für jede gehostete DB
  5. 22:33 — tm.start(blocking=true) → ~6 Min. abwarten
  6. 22:40 — fmaker.stop()
  7. 22:41 — system.update_deploy(label) — Touch-ID-Bestätigung, Robert legt am iPad den Finger auf den Power-Button
  8. ~22:55 (nach Reboot, launchd fährt alles wieder hoch) — fmaker.start()
  9. 23:00 — fmaker.status() + fmaker.list_files() → Hosting verifizieren
  10. 23:01 — logs.tail(service="fms", lines=50) → auf Fehler prüfen
  11. 23:02 — Claude postet in Roberts Chat: "Server aktualisiert auf 15.4.1, FileMaker grün, alle 3 Datenbanken gehostet, letztes Backup 22:32 verifiziert, keine Fehler im Event-Log seit Neustart."

Elf typisierte Aufrufe, ~30 Minuten verstrichen, Robert sieht die grüne Meldung von der Küche aus und geht ins Bett.

Implementations-Fahrplan — was wir bauen, was bei Ihnen installiert wird

Die Auftrennung ist nicht akademisch. Oskar-Seite ist alles, was wir bei uns vorbereiten, codieren und Ihnen als laufenden Dienst übergeben — einmal aufgesetzt, fährt es zwischen Ralphs Besuchen autonom weiter. Oppliger-Seite ist alles, was einmalig auf dem Mac mini installiert und konfiguriert wird, plus die Hardware, die Sie kaufen. Nach dem Setup-Tag sieht Robert weder das eine noch das andere — er sieht nur den Chat.

Das Paradigma in einem Satz

Das hier ist macOS Unix mit FileMaker im Zentrum, nicht x86 Linux mit Postgres im Zentrum. Die ganze Linux-Sysadmin-Liturgie — Prometheus-Dashboards, MDM-Enrollment, fail2ban-Regeln, OpenTelemetry-Pipelines, Docker-Compose-Stacks, dedizierte Service-User, VPN-Mesh, Hardware-Token-Rituale — ist hier Ballast. Sie löst Probleme, die diese Praxis nicht hat, mit Werkzeugen, die niemand in einer Vier-Personen-Praxis pflegen kann. Stattdessen: launchd, Caddy als einziger öffentlicher Listener, FileMaker als Datenbank, Touch ID am Browser, macOS Unified Log als Audit-Spur. Jedes Stück ist schon im macOS oder ein einzelnes Binary mit einer einzigen Konfigdatei — kein Cluster, keine Helm-Charts, kein Service-User-Zoo, keine Mesh-VPN-Enrollment-Tabelle. Heavy-Duty-FileMaker im Kern, leichtgewichtige Peripherie aussenrum — servicierbar durch einen einzigen Menschen, der gelegentlich vorbeischaut. Das ist das Designziel und der gesamte TCO-Vorteil.

WP-S1 · Oskar-Deployment-Bundle — ein Skript, ein Reboot

Oskar-Seite (wir liefern)Oppliger-Seite (einmalig installiert)
Ein einziges oskar-deploy.sh. Idempotent. Erkennt vorhandenes Setup, ergänzt nur das Fehlende. Schreibt die LaunchDaemon-plist nach /Library/LaunchDaemons/ch.oskar.app.plist (läuft als Root, wie jeder andere macOS-Systemdienst — kein dedizierter Service-User-Zoo), holt Code aus dem Git-Repo, baut, startet. Plus ein oskar self-update-Befehl: ab dann zieht Oskar Updates selbst, ohne dass jemand auf den Mac mini muss. Mac mini mit aktuellem macOS, Homebrew einmal installiert, ein git clone, dann sudo ./oskar-deploy.sh. SMS an Ralph für die Einrichtungs-Bestätigung. Fertig. Updates passieren danach autonom — weder Robert noch Bettina müssen je wieder ein Terminal öffnen.

Aufwand: ~2h Claude · ~15 Min Robert am Setup-Tag · 0.5 Tag wall-clock

WP-S2 · Webserver — Caddy als einziges Frontend, plus eine Tunnel-Alternative

Oskar-SeiteOppliger-Seite
Caddy — ein einziges Go-Binary, eine Caddyfile, automatische ACME-Erneuerung, ein launchd-Eintrag. Zwei öffentliche Routen (oppliger.ch → Live-Next.js auf :3000, stage.oppliger.ch → Stage-Next.js auf :3001) plus der Admin-Endpoint auf derselben Caddy-Instanz, geschützt durch Touch ID am Browser. Kein nginx, kein Apache, kein Apache-vs-PHP-FPM-Konfigurationszoo, kein separater Admin-VPN — das gehört in die Linux-Welt und hat hier nichts zu suchen. brew install caddy. Eine Caddyfile-Kopie an den Standardort. Domain-A-Record auf die feste IP des Anschlusses zeigen lassen (ist bereits konfiguriert für FileMaker-Fernzugriff). Eine Port-Forwarding-Regel im Router (80, 443) — oder die Alternative weiter unten.
Alternative falls die feste IP wegfällt oder Port-Forwarding ungewünscht ist: cloudflared als Tunnel-Daemon, ebenfalls launchd-managed. Keine offenen Ports am Router, kein Zertifikatsmanagement, kein DDNS, keine öffentliche IP nötig. Cloudflare terminiert TLS am Edge, der Mac mini bleibt aus dem öffentlichen Internet schlicht unsichtbar. Empfohlen für den Fall, dass die Internet-Verbindung gewechselt wird. Cloudflare-Account (kostenlos für diesen Use-Case), eine Tunnel-ID, ein Zugangstoken, ein brew install cloudflared. Das war's. DNS wird automatisch konfiguriert.

Aufwand: ~1h Claude · DNS-Konfig vorab via Registrar · 0.5 Tag wall-clock

WP-S3 · Auto-Start beim Server-Neustart — launchd, nichts sonst

Oskar-SeiteOppliger-Seite
LaunchDaemon-plist pro Dienst mit RunAtLoad=true und KeepAlive=true (Oskar selbst, Caddy, FileMaker Server). Das ist macOS-natives Cron + Init in einer Datei pro Dienst. Crash → launchd startet neu. Reboot → alles fährt selbst hoch. Kein systemd, kein supervisord, keine docker-compose.yml. Drei plists insgesamt, alle aus dem Setup-Skript installiert. Keiner. plists werden vom Setup-Skript an die Standardpfade geschrieben. Auch wenn der Strom über Nacht ausfällt und der Mac mini neu hochfährt, ist eine Minute später alles wieder online — bevor Robert um 07:30 das Praxistor öffnet.

Aufwand: ~0.5h Claude · inkl. in WP-S1-Deploy · keine Wall-Clock

WP-S4 · Notfall-Doppelklick — ein .app-Bundle für den Fall der Fälle

Oskar-SeiteOppliger-Seite
Ein Oskar-Rettung.app-Bundle (eigentlich ein Apple Shortcut, doppelklickbar von jedem Laptop und vom Empfangs-iPad). Ablauf: launchctl kickstart -k system/ch.oskar.app → 10 Sekunden warten → HTTP-Check auf https://oppliger.ch/_health. Erfolg → macOS-Benachrichtigung «Oskar wieder online». Misserfolg → Dialog mit zwei Buttons: Robert (im Haus) anrufen, Stecker ziehen / Ralph anrufen. Beide öffnen Telefon.app mit der entsprechenden Nummer vorgewählt. Keiner. .app-Bundle wird durch das Setup-Skript in /Applications/ abgelegt, automatisch auf jeden Laptop via iCloud-Sync verteilt. Robert lernt eine Sache: Wenn etwas nicht geht, Oskar-Rettung doppelklicken. Kein Konzept dahinter zu verstehen.

Aufwand: ~1h Claude · 0.5 Tag wall-clock

WP-S5 · macOS- und Paket-Updates — Oskar erledigt sie selbst

Oskar-SeiteOppliger-Seite
Drei Befehlswege (softwareupdate -i -a für macOS, brew update && brew upgrade für Homebrew, mas upgrade für Mac-App-Store-Apps) unter einem MCP-Werkzeug system.update_deploy() gebündelt. Das Werkzeug fragt zuerst fmaker.list_clients() ab, wartet auf das Idle-Fenster, fährt das 11-Schritte-Update aus dem Beispiel oben. Robert tippt «update heute Nacht» in den Chat; Oskar tut den Rest. Automatischer Wartungs-Cron via launchd timer — einmal pro Woche, dienstags 23:00, mit Opt-out-Befehl. Keiner. Robert sieht nur die grüne Bestätigungs-Nachricht am nächsten Morgen. Keine WSUS, keine MDM-Policy, keine Apple-Configurator-Profile.

Aufwand: ~1h Claude · 0.5 Tag wall-clock

Auth-Schicht — ein Satz

Touch ID am Browser bestätigt sowohl den Login als auch jede destruktive Operation — die einzige Auth-Schicht, die Oskar braucht. macOS Platform Authenticator ist schon auf jedem Mac und iPad eingebaut, phishing-resistent, null Extra-Hardware. sudo spielt keine Rolle, weil Oskar als LaunchDaemon ohnehin unter Root läuft — der menschliche Bestätigungsschritt sitzt im UI, nicht in der Shell. Pro-User-ACL: Robert und Bettina haben volle Rechte, Gabriela schreibt nur in ihren TCM-Bereich, MPAs sehen nur Buchung und Recall. Audit-Trail im macOS Unified Log (Subsystem ch.oskar) — jeder Werkzeugaufruf vom OS signiert, durch Time Machine konserviert, abfragbar via log show.

UX-erhaltende Sicherheitshärtung — sieben Massnahmen, alle macOS-nativ

Die Linux-Sysadmin-Tradition produziert lange Checklisten, die kollektiv eine Praxis lahmlegen: erzwungene Passwort-Rotation, MDM-Enrollment, OpenTelemetry-Pipelines, fail2ban mit IP-Reputation-Listen, dedizierte Service-User, Hardware-Token-Rituale, VPN-Meshes mit Enrollment-Tabellen, USV-Wartung. Das alles kostet UX und löst Probleme, die diese Praxis nicht hat. Diese Liste ist anders: jede Massnahme ist entweder schon in macOS eingebaut oder ein einzelner Konfigschritt. Jede ist entweder unsichtbar oder UX-verbessernd. Wenn sie das nicht ist, steht sie nicht hier.

Caddy als einziger öffentlich erreichbarer Dienst; alles andere blockiert durch Router-NAT + macOS Application Firewall (System Settings → Network → Firewall, «Allow specific apps»-Modus) Eine Angriffsfläche, ein Patch-Punkt. Router-NAT ist der primäre Perimeter, der ist schon da. Unsichtbar

Bewusst nicht auf der Liste — was wir gestrichen haben und warum

  • Dediziertes oskar-Servicekonto. Linux-Tradition aus Multi-Tenant-Serverumgebungen. Auf einem Mac mini mit zwei Usern Theater — isoliert Oskar von nichts, was er für seine Aufgabe ohnehin anfassen muss. Oskar läuft als LaunchDaemon unter Root, wie jeder andere macOS-Systemdienst.
  • Tailscale / Admin-VPN-Mesh. Die Praxis hat schon eine feste IP für FileMaker-Fernzugriff — das ist das bewährte Muster. Caddy serviert die Admin-UI öffentlich; Touch ID schützt sie. Wer die Login-Seite sieht, hat nichts gewonnen. Für Ralphs Service-Calls reicht macOS Screen Sharing via iCloud, eingebaut, end-to-end-encrypted.
  • YubiKey für destruktive Ops. Vier Hardware-Schlüssel zu verwalten, lost-key-Recovery zu pflegen, MPA-anruft-Ralph-weil-sie-ihren-nicht-findet-Ritual — gegen einen Nutzen, den Touch ID am Browser kostenlos liefert. macOS Platform Authenticator (Touch ID / Face ID) ist phishing-resistent und schon auf jedem Gerät.
  • Apple Remote Desktop deaktivieren. ARD/Screen Sharing hat macOS-managed Auth pro Session. Deaktivierung kostet Ralphs Fähigkeit, Robert remote über die Schulter zu schauen, und schützt vor nichts Realem.
  • Eigene Snapshot-Schicht in /var/oskar/snapshots/. APFS macht stündliche Snapshots automatisch. Time Machine konserviert die letzten 24 Stunden. FileMaker hat eigene Backups. Eine vierte parallele Backup-Schicht ist Linux-Hausmeisterei.
  • Lokale Audit-Logs + nächtliches Backblaze-Upload. macOS Unified Log existiert schon, ist OS-signiert, durch Time Machine konserviert. Eine separate Log-Rotation-Pipeline mit B2-Upload pflegt sich nicht selbst.
  • 5-Min-Heartbeat via Pushover. Wenn Oskar tot ist, merkt Robert das beim ersten Chat-Versuch. 288 Pings pro Tag mit 99,9 % grünem Punkt sind Aufmerksamkeitsmüll — und liefern False Positives um 3 Uhr morgens.
  • USV. M-Series Mac mini auf APFS verkraftet Stromausfälle problemlos. Switzerland hat < 30 Minuten Haushalts-Stromausfall pro Jahr. CHF 150–300 Hardware + Batteriewechsel alle drei Jahre + ein neuer Monitoring-Punkt für ein Problem, das hier nicht existiert. CHF 30 Steckdose mit Überspannungsschutz reicht.
  • Shelly Plug S für Remote-Power-Cycle. Robert wohnt an derselben Adresse wie die Praxis. Eine Treppe runter, Stecker ziehen — das ist die Recovery-Operation, die einmal alle paar Jahre nötig wird. Smart-Home-Hardware löst hier ein Distanzproblem, das nicht existiert.
  • Mosyle/Jamf MDM-Enrollment, Prometheus + Grafana, OpenTelemetry-Pipelines, fail2ban, Docker / Kubernetes, SIEM / EDR. Linux-Serverraum-Folklore. Eine Vier-Personen-Praxis pflegt keine dieser Werkzeuge; die Probleme, die sie lösen, existieren in dieser Umgebung nicht.

Der Rahmen

Die richtige Lesart von "was Claude kann / nicht kann" ist nicht "wie sperren wir es aus allem Gefährlichen aus"; sondern "wie geben wir ihm die echte Breite eines Admin-Jobs, mit jeder Aktion im macOS Unified Log audit-geloggt und jeder destruktiven Op am Browser per Touch ID bestätigt." Kein Datei-Pfad-Quarantänelager. Kein Sicherheits-Theater. Keine Hardware-Token-Rituale. Die Grenze liegt in der Werkzeug-Semantik und im Audit — nicht in den Verzeichnissen, die Oskar sehen kann.

Das Erlebnis aus Roberts Sicht: "Ich gehe nicht mehr in den Keller. Bettina loggt sich morgen ein und sieht, was passiert ist. Das Team hat einen einzigen Weg, dem Server eine Aufgabe zu geben, und der Server antwortet auf Deutsch."
