Healthcare-specific UX patterns that matter in Switzerland

  • “Accepting new patients?” on the homepage. OneDoc's interface explicitly supports the toggle; on its info site OneDoc states: “When setting up your account, you can choose to restrict online booking to your existing patients … on OneDoc.ch, patients can filter their search to see only general practitioners who are accepting new patients—your profile will not appear in those results.” Mediocre sites force a phone call.
  • Notfall hierarchy — 144 (national), the cantonal Ärztefon (in Zürich: AERZTEFON AG), and the practice's own after-hours rotation must be visually distinct from regular booking.
  • Berufsgeheimnis-aware communication — booking flows must avoid collecting diagnostic data in plain text. Vendors like OneDoc, Arzt-direkt and Doctena limit free-text fields and encrypt clinical data end-to-end; many bespoke sites leak this through unencrypted contact forms.
  • HIN email — Swiss doctors run professional email through hin.ch (Health Info Net AG). Dr. Oppliger correctly uses robert.oppliger@hin.ch. Top sites expose this rather than @gmail.com.
  • Hausarztmodell visibility — patients on this insurance model pay reduced premiums but must use their assigned GP first. MüllerPraxis explains it cleanly: “If you are insured under the mediX family doctor model, we are your primary care provider. We handle referrals to specialists online, so you do not need a paper referral. … your insurer may issue violation notices if you visit another doctor directly (except for eye care and gynecology) without discussing it with us in advance.” That's the gold standard for plain-language insurance UX.

Swiss-specific compliance signaling

nDSG / revDSG — in force 1 September 2023. Fines up to CHF 250,000, personally chargeable to the responsible natural person (the doctor as managing director), not the company. The site's privacy policy must list data categories, retention, recipients, and any cross-border transfer.
Berufsgeheimnis (StGB Art. 321) — per the Kanton St. Gallen guidance (sg.ch/gesundheit-soziales/gesundheit): “Nach Art. 321 des Schweizerischen Strafgesetzbuches … werden Ärzte, Zahnärzte, Chiropraktoren, Apotheker, Hebammen, Psychologen, Pflegefachpersonen, Physiotherapeuten, Ergotherapeuten, Ernährungsberater, Optometristen, Osteopathen sowie ihre Hilfspersonen, die ein Geheimnis offenbaren, das ihnen infolge ihres Berufes anvertraut worden ist oder das sie in dessen Ausübung wahrgenommen haben, auf Antrag mit Freiheitsstrafe bis zu drei Jahren oder Geldstrafe bestraft.” The dominant academic reading (Wohlers, 2016, “Auslagerung einer Datenbearbeitung und Berufsgeheimnis (Art. 321 StGB)”; Rosenthal, “Mit Berufsgeheimnissen in die Cloud”, Jusletter 10 August 2020) extends “Hilfsperson” to IT and cloud providers when objectively necessary to the practice. This is the legal hook for OneDoc, Arzt-direkt — and a future Oskar deployment.
Cantonal requirements — Kanton Zürich publishes a “Merkblatt Berufsgeheimnis und Datenschutz” for medical practices (zh.ch). Any production site should align with it.
Swiss data residency — top-tier vendors lean on this hard. It's a defensible differentiator vs US-hosted alternatives subject to the CLOUD Act.

The five architectural decisions

1 · Oskar runs as an unprivileged LaunchDaemon, not root

Apple's official launchd documentation states the distinction directly: “The main difference is that an agent is run on behalf of the logged in user while a daemon runs on behalf of the root user or any user you specify with the UserName key. Only agents have access to the macOS GUI.” (developer.apple.com / launchd.info.) A server process should be a LaunchDaemon under a dedicated _oskar service account specified via the UserName key, never as root. Privilege escalation happens through a tightly scoped sudoers rule (see #4).

2 · Public traffic and admin traffic segregated at the reverse proxy

Caddy as the front handles auto-HTTPS for public hostnames. Per Caddy's official documentation (caddyserver.com): “Caddy uses HTTPS for all sites by default, as long as a host name is provided in the config … Caddy will attempt to get a publicly-trusted certificate; make sure your DNS records point to your machine and that ports 80 and 443 are open to the public and directed toward Caddy.” The admin plane is exposed only over Tailscale's 100.x.x.x mesh address with mTLS on top — public internet cannot reach admin.oskar.local at all.

example.com {
    reverse_proxy localhost:9000      # public Oskar app
}

admin.oskar.local {
    bind 100.x.x.x                    # Tailscale interface only
    tls /etc/oskar/mtls/cert.pem /etc/oskar/mtls/key.pem {
        client_auth {
            mode require_and_verify
            trusted_ca_cert_file /etc/oskar/mtls/ca.pem
        }
    }
    reverse_proxy localhost:9001      # MCP admin server
}

3 · Claude never gets a shell — it calls a typed MCP tool surface

The Model Context Protocol was launched by Anthropic in November 2024. Per Anthropic's launch announcement (anthropic.com/news/model-context-protocol): “Today, we're open-sourcing the Model Context Protocol (MCP), a new standard for connecting AI assistants to the systems where data lives, including content repositories, business tools, and development environments.” The protocol then saw industry-wide adoption: per OpenAI CEO Sam Altman in March 2025, “People love MCP and we are excited to add support across our products”; Google DeepMind CEO Demis Hassabis confirmed support in April 2025, calling MCP “a good protocol and it's rapidly becoming an open standard for the AI agentic era.” The MCP server should expose only a finite allowlist of typed operations:

  • service.restart(name) — restart Oskar, Caddy, or the database
  • disk.usage() — return df output
  • logs.tail(service, lines) — recent log lines (patient-data fields redacted)
  • certificate.expiry() — TLS certs and renewal dates
  • update.deploy(version) — signed Oskar update from a release channel
  • backup.run() — trigger Time Machine snapshot
  • health.summary() — uptime, CPU, memory, request rates
  • firewall.status() — currently open ports

What is not exposed: arbitrary shell, filesystem reads outside /Library/Logs/Oskar/, any access to /Users/, any access to a patient database, any ability to disable the firewall or SIP. Per GitHub's MCP security analysis: “Exposing internal tools and infrastructure to AI systems, even through structured protocols, introduces new cyberthreat surfaces … MCP deployments typically rely on strong authentication, authorization, auditing, and execution boundaries implemented by the client or host environment.”

5 · Every Claude action is logged immutably

The MCP tool dispatcher writes an append-only audit record before every call: timestamp, authenticated identity, tool name, parameters (sensitive args redacted), result hash, exit code. Logs ship via OpenTelemetry to an external EU-hosted sink (Grafana Cloud EU, Axiom EU, or self-hosted Loki) so that even if the Mac mini is compromised, the audit trail survives. This satisfies the nDSG Art. 8/9 data-security obligation. Per the Kiteworks Swiss healthcare compliance guide: “FADP Article 8 requires special safeguards for health data, while Article 9 mandates data security preventing unauthorized access, with violations creating financial penalties up to CHF 250,000.”

Sandboxing options ranked

  1. Best — Oskar runs as a non-root LaunchDaemon under _oskar, with TCC permissions denied, sudoers narrowly allowlisted, and the MCP admin surface bound to tailscale0 only.
  2. Acceptable — Oskar runs inside OrbStack or Lima (lightweight Linux VMs on macOS) — adds a kernel boundary but adds ARM/Linux compatibility surface.
  3. Not recommended — Docker Desktop with bind mounts into /Users/ — too easy to misconfigure and accidentally expose patient-adjacent paths.
What must NEVER be exposed to Claude
  • Any directory under /Users/ (especially ~/Documents, ~/Desktop, ~/Downloads)
  • The practice management software's database files. Per OneDoc's compatibility documentation: “As a general practitioner, you may already use software like MediOnline, Mediway, or Axenita to manage your practice.” These contain patient health data protected by Berufsgeheimnis Art. 321 StGB.
  • The HIN email client and its local cache
  • The Keychain (passwords, certificates)
  • The Time Machine backup volume contents
  • Anything matching *.dcm (DICOM imaging), *.hl7, or *.pdf in user home directories
  • Disable-or-modify operations on XProtect, Gatekeeper, FileVault, the firewall, SIP, MDM enrollment

The compliance angle, plain-stated

Swiss medical-practice data sits under two parallel regimes:

  1. nDSG / revDSG (federal data protection, in force 1 September 2023). Per Härting Rechtsanwälte's commentary: “With the new Data Protection Act, fines of up to CHF 250,000.00 can be imposed for violations pursuant to Art. 60-66 FADP.” These are personally chargeable to the responsible natural person, not the company.
  2. Berufsgeheimnis (StGB Art. 321) — criminal liability (up to three years' imprisonment or monetary penalty) for unauthorized disclosure of patient information by doctors AND their “Hilfspersonen.” Per Rosenthal (2020), the test is: “Damit es durch den Beizug des Dritten nicht zum tatbestandlichen Offenbaren eines Geheimnisses kommt, muss der Beizug … erstens zur Erbringung der Dienstleistung erforderlich sein und zweitens für den Geheimnisherrn … voraussehbar sein.” IT and cloud providers fall within this when objectively necessary to the practice.

The architecture above keeps Oskar firmly outside the patient-data perimeter. Oskar handles website + booking only, and the booking data it does collect (name, contact, requested slot) is the same minimum set OneDoc / Doctena / Arzt-direkt collect — hosted on the Mac mini in Switzerland, with end-to-end TLS and audit logging. The Claude-administered admin plane only ever touches infrastructure — never the booking database content, never patient communications — and this boundary is enforced architecturally, not just by policy.

The concrete switch-on checklist

  1. Create a dedicated _oskar service account on the Mac mini; no GUI login.
  2. Install Caddy via Homebrew; configure as LaunchDaemon with the Caddyfile pinned to public hostnames + Tailscale-only admin.
  3. Install Tailscale; enrol the Mac mini; tag tag:oskar-prod. Issue machine-scoped auth keys for CI.
  4. Configure pf to permit only 80 / 443 on the public interface and only specific ports on tailscale0.
  5. Disable Apple Remote Management (ARD); disable public SSH; enable SSH only on tailscale0.
