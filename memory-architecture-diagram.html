1

CLAUDE.md — Static Identity

User-written, never auto-modified. Discovered by walking up filesystem.
EXISTS ▶
CLAUDE.md → walkUp(cwd→root) → dedup by hash → System prompt injection

Budget

4,000 chars/file, 12,000 chars total. Checks CLAUDE.md, CLAUDE.local.md, .claude/CLAUDE.md at each directory level.

Role

Identity. "I am a Python project. We use tabs. The CEO is Sarah." Config file, not architecturally interesting.

2

memdir/ — Persistent Topic Files

Filesystem-as-database. MEMORY.md index + topic files with YAML frontmatter.
PORT ▶
~/.claude/projects/{git-root}/memory/ → MEMORY.md (index, 200 lines max) + topic-files.md (frontmatter: name, desc, type)

Path Resolution

  • CLAUDE_COWORK_MEMORY_PATH_OVERRIDE env var
  • autoMemoryDirectory in settings.json
  • Default: ~/.claude/projects/{sanitized}/memory/

Security: rejects relative, root, UNC, null-byte paths. Excludes projectSettings to prevent malicious repo exploits.

4-Type Taxonomy

  • user — role, preferences, style
  • feedback — corrections + confirmations
  • project — ongoing work, deadlines
  • reference — external system pointers

Scanning (memoryScan.ts)

Recursive readdir → filter .md (exclude MEMORY.md) → parallel frontmatter read → sort newest-first → cap at 200. Single-pass stat+read halves syscalls.

Recall (findRelevantMemories.ts)

Scan → filter alreadySurfaced → Sonnet sideQuery() picks ≤5 files → inject into system prompt. Only place that uses cheaper model.

3

Session Memory — Within-Session Notes

10-section structured notes. Fires as postSamplingHook. Dies with session.
FREE IN BRIDGE ▶
postSamplingHook → shouldExtractMemory() → runForkedAgent() → session-memory.md

Trigger Thresholds

  • Init: 10,000 tokens in context
  • Update: 5,000 token growth since last
  • Tool calls: 3 since last extraction
  • Safe point: no active tool calls

10 Sections

Session Title, Current State, Task Spec, Files & Functions, Workflow, Errors, Codebase Docs, Learnings, Key Results, Worklog. 2K tokens/section, 12K total.

4

Extract Memories — Between-Turn Promoter

Background agent after each turn. Promotes signal from session to disk (Layer 2).
ADAPT ▶
stopHooks (post-turn) → cursor check (lastMemoryMessageUuid) → hasMemoryWritesSince? skip : run → runForkedAgent() (Sonnet) → memory/*.md

Coalesced Execution

inProgress flag + pendingContext stash + trailing run in finally block. At most 2 extractions in flight. No backlog, no starvation.

Efficiency Instructions

Turn 1: all FileRead calls in parallel.
Turn 2: all FileWrite/Edit calls in parallel.
No interleaving. maxTurns: 5 hard cap.

Cursor System

lastMemoryMessageUuid — only processes messages after this UUID. Advances after successful run. Stays put on error (retry).

Mutual Exclusion

If main agent already wrote to memory dir → skip that range. Main agent's writes are authoritative.

Four-Phase Consolidation Prompt

Phase 1

Orient

ls memory dir, read MEMORY.md, skim topic files. Know what you know.

Phase 2

Gather

Daily logs → drifted memories → narrow transcript grep. Don't read wholesale.

Phase 3

Consolidate

Merge into existing files. Convert relative dates. Delete contradictions.

Phase 4

Prune

MEMORY.md under 200 lines. ~150 chars/entry. Remove stale pointers.

Lock File (.consolidate-lock)

Triple duty: mtime = lastConsolidatedAt, body = holder PID, rollback via utimes(). Stale after 60min even if PID live (reuse guard).

Execution

runForkedAgent() shares parent prompt cache via cacheSafeParams. Progress tracked by makeDreamProgressWatcher(). Rollback on failure, scan throttle = backoff.

3

Session Files — Within-Session State

SESSION.md + IMAGES.md + BUILD.md + CREATIVE-BRIEF.md. Written by CD agent.
EXISTS ▶
CD Agent (bridge) → public/{sessionId}/SESSION.md + IMAGES.md + BUILD.md + CREATIVE-BRIEF.md

Why we don't need Session Memory (Layer 3 from Claude Code)

Bridge mode runs Claude CLI as a persistent subprocess. The CLI already has its own session memory built in. Plus, OskarOS has structured session files that serve the same purpose. Don't rebuild what's free.

4

Extract Memories — Post-Turn Promoter

After each bridge response, promotes signal to memory/. Fire-and-forget.
BUILD ▶
Bridge 'result' event → extractMemoriesIfNeeded() → coalesced execution → Anthropic API (Sonnet) → memory/*.md

Coalesced Execution (ported)

Same pattern: inProgress + pendingContext + trailing run. Max 2 extractions in flight. Recent messages passed as string (no cursor UUID needed — bridge gives us the exchange directly).

Integration Point

In chat-stream route, after yielding final result event:
extractMemoriesIfNeeded(messages).catch(…)
Fire-and-forget. Don't block response.

Four-Phase Prompt (adapted for session files)

Phase 1

Orient

Read memory/MEMORY.md index. Skim existing topic files. Know what you know.

Phase 2

Gather

Read SESSION.md, IMAGES.md, BUILD.md, CREATIVE-BRIEF.md from each session. Structured markdown, not transcript grep.

Phase 3

Consolidate

Merge cross-session patterns. Convert dates. Delete contradictions. Use FileEdit on existing files.

Phase 4

Prune

MEMORY.md under 200 lines. ~150 chars/entry. Remove stale. Resolve conflicts.

API Route

POST /api/dream — check gates + dream
POST /api/dream?force=true — bypass gates
GET /api/dream — gate status check
Returns: { dreamed, reason, filesWritten, turns }

Execution

Direct Anthropic API loop with Sonnet. Custom tool set (FileRead, FileWrite, FileEdit, Glob, Grep). Split permissions: READ from public/, WRITE to memory/ only. Max 10 turns.

Lock File (identical to Claude Code)

memory/.consolidate-lock — mtime = lastConsolidatedAt, body = PID, stale after 60min, rollback via utimes(). Race resolution: last writer wins.

Key Adaptation

Phase 2 reads structured session files directly instead of grepping JSONL transcripts. Simpler AND richer signal — files are already organized by concern.
